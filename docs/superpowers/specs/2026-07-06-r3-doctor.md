# R3 spec — `nowdocs doctor` diagnostics

> **Parent track:** `docs/superpowers/specs/2026-07-06-robustness-ux.md`
> **Implementation plan:** `docs/superpowers/plans/2026-07-06-robustness-ux-plan.md`
> **Depends on:** R2 staging/rollback conventions
> **Stage:** R3 — read-only diagnostics and MCP smoke
> **Difficulty:** 5/10 (medium). Main review risk is user-facing accuracy and not mutating cache state.
> **Worktree rule:** implement from a dedicated git worktree and branch for R3, e.g. `../nowdocs-r3-doctor` on `feat/r3-doctor`.

---

## 0. Goal

R3 adds `nowdocs doctor`, a read-only diagnostic command that tells users whether their nowdocs environment, cache, installed docsets, model files, and MCP setup are healthy.

R3 should make failures actionable without deleting or rewriting user data. Repair is limited to explicit future R4 behavior; R3 may reject `--repair` with a clear message if R4 is not implemented.

---

## 1. Non-goals

- Do not implement transactional install/update; R2 owns that.
- Do not implement cache cleanup/repair beyond a clear unsupported or not-yet-implemented message.
- Do not implement `smoke`; U1 owns user-facing retrieval smoke.
- Do not add network server, host/port, or HTTP MCP transport.
- Do not add telemetry or remote diagnostics.

---

## 2. Dispatch metadata

- **Estimated difficulty:** medium, 5/10.
- **Expected implementation time:** 1 day for a Rust maintainer; 1-2 days for an external coding agent including JSON tests and CLI tests.
- **Review risk:** medium. Focus on exit codes, JSON shape, clear remediations, no cache mutation, and compatibility with R2 staging paths.
- **Required isolation:** independent git worktree and stage-specific branch. Do not share a dirty worktree with R2/U1.

---

## 3. CLI contract

Add:

```bash
nowdocs doctor [--json] [--docset <name>] [--mcp] [--model] [--repair]
```

Semantics:

- default is read-only checks.
- `--json` emits experimental JSON until v1.0.
- `--docset <name>` deep-checks one installed docset.
- `--mcp` runs an in-process initialize/tools-list style smoke without external client or network.
- `--model` checks model cache state; it should not force model download unless a future flag explicitly does that.
- `--repair` is not R3 unless R4 has landed; return a clear not-yet-implemented message or no-op with warning if product direction changes.

Exit codes:

| Aggregate status | Exit code |
|---|---|
| ok | 0 |
| warn only | 0 |
| any fail | 1 |

---

## 4. Check model

Each check should have:

- `id`
- `severity`: `ok`, `warn`, or `fail`
- `message`
- `remediation`

Top-level JSON shape:

```json
{
  "status": "ok|warn|fail",
  "checks": []
}
```

The top-level shape should remain stable; extra fields may be experimental.

---

## 5. Default checks

Implement at least:

- binary/version/platform basic info.
- cache root exists or can be created.
- cache root is writable.
- expected cache directories exist or can be created.
- installed docset manifest/store presence consistency.
- manifest parses and validates.
- stale staging/rollback dirs detected using R2 conventions.

No default check may require network.

---

## 6. `doctor --docset`

For one docset:

- validate docset name.
- check manifest path.
- parse/validate manifest.
- check store path.
- try opening the store if cheap and deterministic.
- check source license/notice paths if relevant.
- print actionable remediation if missing/corrupt.

---

## 7. `doctor --mcp`

Run a local, in-process MCP smoke:

- initialize capability check, or direct handler-level equivalent.
- tools/list includes expected tools.
- must not spawn a long-running external MCP client.
- must not bind network ports.

---

## 8. Required tests

- CLI help lists `doctor` and flags.
- clean temp cache reports ok/warn but no fail.
- missing/corrupt manifest reports fail and non-zero exit.
- `--json` output parses and has `status` + `checks`.
- `--docset` validates docset name.
- stale staging path is reported.
- `--mcp` succeeds without network.
- `--repair` behavior is explicit and non-destructive.

---

## 9. Verification commands

```bash
cargo test doctor_tests -- --test-threads=1
cargo test cli_tests -- --test-threads=1
cargo fmt --check
git diff --check
```

If no separate `doctor_tests.rs` is created, run the relevant CLI/integration test file and explain where doctor tests live.

---

## 10. Done definition

R3 is done when:

- users can run `nowdocs doctor` on a clean environment.
- JSON diagnostics are parseable.
- docset/cache failures are classified with remediation.
- MCP setup can be sanity-checked without network.
- no diagnostic path deletes or rewrites active user data.
