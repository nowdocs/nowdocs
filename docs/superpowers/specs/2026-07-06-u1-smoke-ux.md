# U1 spec — `nowdocs smoke` and first-run output UX

> **Parent track:** `docs/superpowers/specs/2026-07-06-robustness-ux.md`
> **Implementation plan:** `docs/superpowers/plans/2026-07-06-robustness-ux-plan.md`
> **Depends on:** R2 fail-safe install/update; benefits from R3 diagnostics but can be implemented after R2 if needed
> **Stage:** U1 — post-install smoke search and clearer success output
> **Difficulty:** 4/10 to 5/10 (medium). Main review risk is first-run model behavior and stable CLI output tests.
> **Worktree rule:** implement from a dedicated git worktree and branch for U1, e.g. `../nowdocs-u1-smoke-ux` on `feat/u1-smoke-ux`.

---

## 0. Goal

U1 gives users a simple confidence path after install or ingest:

```bash
nowdocs smoke <docset> [query] [--json] [--top-k <n>]
```

It also improves success output for install/update/ingest/share/list-installed so users see the next useful command instead of terse confirmation only.

---

## 1. Non-goals

- Do not change retrieval ranking logic unless a small formatting wrapper is required.
- Do not implement `doctor`; R3 owns diagnostics.
- Do not implement cache repair or registry discovery.
- Do not change MCP transport or add host/port flags.
- Do not make smoke a shallow structural check; smoke is real retrieval.

---

## 2. Dispatch metadata

- **Estimated difficulty:** medium, 4/10 to 5/10.
- **Expected implementation time:** 0.5-1 day for a Rust maintainer; 1-2 days for an external coding agent if model/cache edge cases require fixes.
- **Review risk:** medium. Focus on CLI compatibility, JSON shape, model download messaging, no network surprise beyond existing embedder path, and tests not becoming flaky.
- **Required isolation:** independent git worktree and stage-specific branch. Do not share a dirty worktree with R2/R3.

---

## 3. CLI contract

Add:

```bash
nowdocs smoke <docset> [query] [--json] [--top-k <n>]
```

Defaults:

- query: `installation configuration example`
- top_k: choose the existing retrieve default or a small explicit default such as 3/5.
- human output by default.
- JSON output when `--json` is set.

Exit behavior:

- invalid docset: non-zero.
- missing docset: non-zero with hint to install/ingest.
- zero results: non-zero with hint to run `doctor --docset` if available.
- successful results: zero.

---

## 4. Real retrieval semantics

`smoke` must run the real retrieval path:

1. validate docset name.
2. verify docset appears installed.
3. embed query through the existing embedder path.
4. run retrieve/search.
5. print top results.

Because this uses the real embedder, first run may load or download the model through the existing model path. Errors must explain this and point to `nowdocs doctor --model` when R3 is available.

---

## 5. Human output contract

Human smoke output should include:

- docset.
- query.
- elapsed time.
- result count.
- per result: rank, score, heading/source URL, chunk index, short text preview.

Example shape:

```text
smoke ok: nextjs-docs
query: How do I configure middleware?
results: 3 in 187ms
1. score=0.92 heading="Middleware" source=https://... chunk=42
   Use middleware to ...
```

---

## 6. JSON output contract

Minimum JSON shape:

```json
{
  "docset": "nextjs-docs",
  "query": "...",
  "elapsed_ms": 123,
  "result_count": 3,
  "results": []
}
```

Each result should include score, heading/source URL, chunk index, and text preview or text depending on existing retrieve output.

---

## 7. Success output improvements

Improve command outputs while preserving existing test substrings when possible:

- install/update: include version/chunk count/license when available and next command `nowdocs smoke <docset> "..."`.
- ingest: include files/chunks, manifest/source/license summary, warnings for missing source metadata, next smoke/share command.
- share: include output path, manifest/chunks presence, no-vector reminder, registry PR readiness hint.
- list-installed: improve human table with docset/version/chunks/license/status, or keep old output behind compatibility if needed.

---

## 8. Required tests

- CLI help lists `smoke` and flags.
- missing docset smoke exits non-zero with hint.
- invalid docset smoke exits non-zero.
- fixture docset smoke returns results.
- `--json` output parses and has required keys.
- install/ingest/share success outputs include next-step hints.
- list-installed output remains deterministic.

Avoid tests that require network. If model loading is expensive in default tests, use an existing fixture/stub pattern or mark a full model smoke as ignored/manual according to project precedent.

---

## 9. Verification commands

```bash
cargo test cli_tests -- --test-threads=1
cargo test retrieve_tests -- --test-threads=1
cargo fmt --check
git diff --check
```

Manual check if feasible: install/ingest a small fixture and run `cargo run -- smoke <docset> "middleware example"`.

---

## 10. Done definition

U1 is done when:

- users can run `nowdocs smoke` after install/ingest and see real retrieval output.
- missing/invalid docsets fail with actionable hints.
- JSON smoke output is parseable.
- first-run model behavior is explained in errors/output.
- success outputs point users to the next useful command.
